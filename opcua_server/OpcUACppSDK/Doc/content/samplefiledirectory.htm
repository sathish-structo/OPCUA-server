FileDirectory

Top  Previous  Next

This sample shows how a server can expose a directory of a file system to OPC UA including files and subdirectories.
The client is able to modify the content of the exposed directory and subdirectories by create, delete, copy, move or rename files or directories and can read and write the content of the files.
The client operations modifies both, the server address space and the content of the directory on the file system.

 

Where to Find

The sample projects are located in the directory: "<InstallDir>/Apps/Samples/Cpp/FileDirectory" and the sources are in the subdirectories "FileDirectoryServer" and "FileDirectoryClient".
The actual files exposed by the server sample are in the "ServerFiles" subdirectory.

 

Server Implementation

The server creates and exposes an initial structure of FileType and FileDirectoryType nodes organized under a "FileSystem" root FileDirectory object.
Initially the directory contains two subdirectories and three files.
For UserAccessDir directory and user.file the directory and file access is granted to authenticated user, while the access is denied for anonymous users (see User Authorization). For the Role based authorization to work, different Roles are assigned to different users upon successful authentication. For this sample the well-known Roles Anonymous and AuthenticatedUser are used.

The Server::File and Server::FileDirectory classes are extended by the custom MyFile and MyFileDirectory.
All predefined and dynamic created instances are instances of these custom classes with the possibility to deny delete, move or rename operations for all users.
The Method calls are forwarded to the base implementation, both the File and FileDirectory classes rely on the local file system.

 

Running the Sample

•The server needs an existing application instance certificate and it must be in the correct directory. See Setting up a Certificate Store for how to create the demo certificate store. The path to the certificate is specified at the top of FileDirectoryServer.cpp, adapt it if it doesn't fit.
•Compile and start the sample application FileDirectoryServer. The server is listening for incoming connections at the address "opc.tcp://localhost:4880".
•Connect to the sample server, either with the FileDirectoryClient or with your favorite OPC UA client (e.g. Softing OPC UA Client), browse the "FileSystem" root object and check the object hierarchy.
You can always rebrowse this node to check the modifications performed by the FileDirectoryClient application.
Use the following authentication settings for different access rights on UserAccessDir and user.file:
oAnonymous:
Has no rights to modify the content of UserAccessDir nor to read or write the content of user.file.
oUserName - "user" - "pwd":
Has full rights to modify UserAccessDir and user.file.
•Call the methods of the FileDirectory or FileType objects to modify the content of the directories and files of the exposed main directory.

 

Points of Interest

•How to represent real-world files and directories in the address space of a server by using the Server::File and Server::FileDirectory classes, organize them in a hierarchical file system (see function createAddressSpace() in FileDirectoryServer.cpp).
•Extend the Creator and the two base classes to customize file system operations (see "preserveLocation" flag and the overridden methods in MyFileDirectory.cpp).
•When testing, have a look on the exposed directories and files to see the modifications.

 

Client Implementation

The client application shows basic operations on the FileDirectoryType and FileType nodes exposed by the server. It is hard coded to use the nodes under the "FileSystem" root FileDirectory object.

For convenience the helper class ClientDirectory was created.
It wraps all low level calls performed on the Client::Session object, providing a good separation between the actual SDK calls and the code written for user interaction.
ClientDirectory instances are bound to a session and a NodeId.
This is the NodeId of a parent FileDirectory object as all UA methods perform actions on the files and directories organized by this object.

For the UA method calls the well known NodeIds defined by FileDirectoryType are used.
The findChildren() and listAll() methods show a simple way to recursively walk through the file system hierarchy and display a list of organized objects.

For the File content handling there are three convenience methods implemented (getFileSize(), readFile() and appendToFile()).
The methods are static as they are called on a File object identified by a NodeId and these operations are not bound to a parent FileDirectory.

 

Running the Sample

•The client expects the server to run at "opc.tcp://127.0.0.1:4880", if this does not match, adapt the global variable g_serverUrl at the top of FileDirectoryClient.cpp.
•Compile and start the sample applications FileDirectoryServer and FileDirectoryClient.
•The available commands are listed by the brief usage message on startup.
•Use the client command "co" followed by a number to connect as a predefined user with a certain user authentication token or as an anonymous user.
•Use the client command "di" to disconnect from the server.
•Use the client command "ls" to list all available FileDirectoryType and FileType nodes under the root "FileSystem".
Available objects are presented as a simple tree with their DisplayName and NodeId.
For clarity the names representing FileDirectories are enclosed in square brackets: "[]". The index numbers in this list are used to identify objects for other commands.
•Use the client command "mkd" to create a new FileDirectory object. You must specify the index of the parent FileDirectory and the name of the new directory.
•Use the client command "mkf" to create a new File object. You must specify the index of the parent FileDirectory and the name of the new file.
•Use the client command "del" to delete a File or FileDirectory.
•Use the client command "cp" to copy a File or FileDirectory object.
You must specify the index of the object to copy, the index of the target FileDirectory and a name for the object's copy.
A copy with the same name is created if no name is specified.
•Use the client command "mv" to move a File or FileDirectory object.
You must specify the index of the object to move, the index of the target FileDirectory and a new name for the moved object.
Object will keep the existing name if no name is specified.
The move command can also be used to rename a File or FileDirectory, simply use the current parent FileDirectory as target FileDirectory.
•Use the client command "rd" to read and display the content of a File object.
•Use the client command "wr" to append some predefined content to an existing File object.

 

Points of Interest

•Recursively walk through the objects organized by a root FileSystem object.
•Call methods to create, delete, copy, move or rename FileType and FileDirectoryType objects and dynamically modify the file system hierarchy.
•Call methods to read and modify files exposed by a server.

Customizing Acknowledgement

Top  Previous  Next

Classes derived from AcknowledgeableCondition (especially alarms) usually need to be get an acknowledgement / confirmation from a client. The UA Methods Acknowledge and Confirm are exposed in the AcknowledgeableCondition for that purpose.

Confirmation is optional. In case no confirmation is required, please remove the optional property and method from the condition using

// remove ConfirmedState

pAlarm->getConfirmedStateVar()->removeTree();

 

// remove Confirm method

pAlarm->getConfirm()->removeTree();

The default handling of acknowledgement and confirmation is implemented in AcknowledgeableConditionAcknowledge::handleExecuteRequest and AcknowledgeableConditionConfirm::handleExecuteRequest.

This implementation can be used if the alarm acknowledge / confirm status is maintained by the server only. In case of maintaining the acknowledge / confirm status in an underlying system, these methods needs to be overwritten.

Ensure in case of overwriting the methods

•The implementation must not use blocking calls, e.g. sending a request to an underlying system and waiting for response.
•The main functionality needs to be implemented in the overwritten method
•retrieve the condition from the event ID
•validate that the condition is not in an invalid state (e.g. already acknowledged)
•set the given comment
•call onAcknowledge / onConfirm on the condition instance (maintains properties retain, acknowledged, ... and reports the changed condition)
•complete the request (send the result to the client)
•optionally report audit events.

 

In case of a necessary communication to an underlying system you have to split the main functionality into two parts, a synchronous and an asynchronous one.

The synchronous part

•retrieves the condition
•stores the request (e.g. using a cookie) to be available when the communication gets a response
•starts communication to the underlying system.

The asynchronous part is invoked by the response of the communication to the underlying system. This part

•finds the matching request (e.g. by the cookie)
•performs the rest of the default implementation (state validation can be skipped assuming this is already done by the underlying system)

Note:

Never forget to complete the request. A pending request will cause a timeout at client side but additionally a leak at server side.

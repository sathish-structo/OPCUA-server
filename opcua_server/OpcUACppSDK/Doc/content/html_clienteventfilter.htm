Creating an Event MonitoredItem

The MonitoredItem can be used to either monitor changes on an attribute of a node (in most cases the value of a Variable) or an event.

For monitoring events you have to

•Specify a NodeId of on object with the SubscribeToEvents bit set in its EventNotifier attribute.
You can specify any node in the servers address space which is connected via a "HasNotifier" reference to the nodes) of interest.
A server reporting events typically has the SubscribeToEnvent bit set at the well-known "Server" node, where all events are reported.
•Specify the EventNotifier attribute as attribute.
•Configure an event filter (at least the select clause).

Typically you should use a sampling interval of 0 (fastest possible) to allow sampling of events occurring at the same time.

A queue size of 0 (default queue size for event monitored items) is also recommended. The server shall use a queue size where a queue overflow is unlikely.

Client::MonitoredItemPtr monitoredItem = Client::MonitoredItem::create();

monitoredItem->setNodeId(monitorNodeId);

monitoredItem->setAttributeId(EnumAttributeId_EventNotifier);  // an event monitored item needs to target the EventNotifier attribute

monitoredItem->setSamplingInterval(0);                         // an event monitored item needs a sampling interval of 0

monitoredItem->setQueueSize(0);                                // let the server choose the appropriate quque size

EventFilter eventFilter;                                       // for events we need an event filter

...                                                            // configure the EventFilter

monitoredItem->setEventFilter(filter);

 

Defining an Event Filter

These steps must be performed to create an EventFilter:

1.Create an EventFilter object.

EventFilter eventFilter;

2.Configure the select clause using the SimpleAttributeOperand class (see SimpleAttributeOperand and EventFilter::addSelectClause() for further reference).
Typically properties like "Message", "SourceNode", "Severity", "EventId", "Retain" or "ActiveState" ("ActiveState"/"Id") are of interest.

SimpleAttributeOperand selectClause;

selectClause.setTypeDefinitionId(Statics::ObjectTypeId_BaseEventType);        // the NodeId of the event type

QualifiedName browseName;

browseName = QualifiedName(BrowseName_EventType, 0);  // the browse path from the type to the desired property

selectClause.addBrowsePath(&browseName);

selectClause.setAttributeId(EnumAttributeId_Value);   // the desired Attribute, here the value of the property

eventFilter.addSelectClause(&selectClause);

...                                                   // add more select clauses

3.Configure the optional where clause as a ContentFilter containing several ContentFilterElements. A ContentFilterElement consists of an operator and one or more operands of various types depending on the operator. (See ContentFilter, ContentFilterElement and see below how to create a ContentFilter)

ContentFilter whereClause;

ContentFilterElement filterElement;

filterElement.setFilterOperator(EnumFilterOperator_OfType);                   // for example we filter for the event type

FilterOperand filterOperand;

LiteralOperand literalOperand;

NodeId desiredEventTypeNode = Statics::ObjectTypeId_AlarmConditionType;       // and all alarm conditions and their subtypes

literalOperand.setValue(Value(&desiredEventTypeNode));

 

filterOperand.setLiteralOperand(&literalOperand);

filterElement.addFilterOperand(&filterOperand);

 

whereClause.addElement(&filterElement);

eventFilter.setWhereClause(whereClause);

...

monitoredItem->setEventFilter(filter);                                        // use the event filter at the monitored item

 

 

Creating a ContentFilter

A ContentFilter can contain several ContentFilterElements. A ContentFilterElement consists of an operator and one or more operands of various types depending on the operator. For each operator a distinct action is taken that works on the evaluation of the operands. The operators are defined in EnumFilterOperator.

An operand can be Literal for values, SimpleAttribute for an Attribute of the event or one of its Properties, or Element to use another operator as operand. The operands are listed in EnumFilterOperand and specify the classes LiteralOperand, SimpleAttributeOperand and ElementOperand. The defined operator "Attribute" is not allowed for Event filter.

The Element operand refers to another ContentFilterElement by its index in the ContentFilter. With this it is possible to create an evaluation tree using more than one operand. Only forward references are allowed to avoid circular references which means that the referenced index must be greater than the index of the actual ContentFilterElement where the ElementOperand resides in.

What is important to keep track while working with the API is that only the first ContentFilterElement and the referenced ones are evaluated. ContentFilterElements that are not connected to the evaluation tree of the first element will be ignored. Also, keep in mind that if the content filter does not represent an evaluation tree, the event MonitoredItem creation in the server will be rejected.

The ContentFilterElement can return the results TRUE, FALSE and in some cases NULL. The NULL result is interpreted as an error. If the root of the evaluation tree (the first ContentFilterElement) returns a NULL value, then the ContentFilter will also return FALSE.

Note:

•The chosen values for the select clause can be any Attribute or Property of an event type. Have a look on the desired event types to see its Attributes and Properties.
•If a selected Property doesn't exist (e.g. only a certain part of the results has this Property) then a null value will be returned for it.
•If the where clause is left empty then all event types are selected.
•The in EnumFilterOperator defined operators InView and RelatedTo are not allowed for event filters.
•When defining a ContentFilterElement, the number of operands must fit to the used operator. Defining less or more will cause errors.

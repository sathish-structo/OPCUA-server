EnumSecurityCheck Reference

Top  Previous  Next

The EnumSecurityCheck defines bits for security check related validations, which can be disabled Disabling any security related validation will introduce a risk and should not be needed, if the corresponding UA application is implemented according to specification and implements all CVE risks.

EnumSecurityCheck_None = 0x0

Enabled all security checks - default.

EnumSecurityCheck_MissingNonce = 0x01

Allows to connect to a server, which does not send a nonce.

Client session specific. Client sessions will use the application setting by default. If only specific sessions shall disable this validation, disable it at the specific session.

This nonce is a random number (at least 32 byte), which has to be generated on every CreateSession and ActivateSession response by the server. The nonce is part of the encrypted part for user validation and validated at server side. When allowing empty or repeated nonces, then the messages are vulnerable for replay attacks.

Note:

Ignoring this validation is a security risk and contradicts to the OPC UA specification. This option shall only be used when dealing with a non compliant server, where the session establishment would fail otherwise.

Using this option will ignore a missing nonce for user token security, missing nonces for message security cannot be ignored. Problems with missing nonces in message security shall be solved by using MessageSecurityMode None.

There is a released CVE (CVE-2019-19135) that this behaviour has high security impacts. If you need to enable this work-around, please contact the vendor of the UA server causing this need. The vendor shall fix the issue as soon as possible - the work-around for this issue may be removed in later releases of the SDK.

EnumSecurityCheck_DuplicateNonce = 0x02

Allows to connect to a server, which does not generate new nonces.

Client session specific. Client sessions will use the application setting by default. If only specific sessions shall disable this validation, disable it at the specific session.

The effects are similar to sending empty nonces (see EnumSecurityCheck_MissingNonce).

Note:

There is a released CVE (CVE-2019-19135) that this behaviour has high security impacts. If you need to enable this work-around, please contact the vendor of the UA server causing this need. The vendor shall fix the issue as soon as possible - the work-around for this issue may be removed in later releases of the SDK.

EnumSecurityCheck_WrongSeqenceNumber = 0x04

Allows to ignore implementation regarding mantis issue (released with Errata 1.04.3)

General communication setting. Can be disabled on complete communication, both for server and client. Needs to be disabled at Application.

The mantis issue contains the problem that messages can be removed during a roll-over of sequence numbers (at 4.294.966.271). The issue was solved by defining a concrete start of sequence numbers (1023) including after roll-over. Previous specifications allowed all starting sequence numbers less than 1024, so the new definition does not contradict the older specifications. For compatibility reasons sequence numbers less than 1023 and greater than 4.294.966.271 shall be still accepted.

Note:

Ignoring the mantis issue will allow the client or server to communicate with non-compliant OPC UA applications. This option shall only be used if dealing with a non-compliant UA applications, where the session establishment would fail otherwise. When ignoring the implementation for the issue, messages can be lost without recognizing.

If you need to enable this work-around, please contact the vendor of the UA application causing this need. The vendor shall fix the issue as soon as possible - the work-around for this issue may be removed in later releases of the SDK.

EnumSecurityCheck_IdenticalEndpointDescription = 0x08

Allows to ignore different EndpointDescriptions

Client session specific. Client sessions will use the application setting by default. If only specific sessions shall disable this validation, disable it at the specific session.

A server returns endpoint descriptions both in the GetEndpoints response as in the CreateSession response. Both endpoint descriptions shall be identical and a client shall validate this. The reason for validation: The GetEndpoints is invoked on a non-secure communication and the response may be altered by man-in-the-middle. By altering the response, either security policies can be removed, endpoint URLs can be changed, ... This may cause connections to a wrong server, using a non-secure policy, ...

Note:

Ignoring the validation will allow the client to communicate with non-compliant OPC UA applications. This option shall only be used if dealing with a non-compliant UA applications, where the session establishment would fail otherwise. When ignoring the validation you introduce security risks.

If you need to enable this work-around, please contact the vendor of the UA server causing this need. The vendor shall fix the issue as soon as possible - the work-around for this issue may be removed in later releases of the SDK.

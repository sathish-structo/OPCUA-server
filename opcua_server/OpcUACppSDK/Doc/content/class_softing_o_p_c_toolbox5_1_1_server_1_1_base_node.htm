virtual EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::init(const INodeId * typeId) [virtual]

Initializes a UA object or variable and creates its sub-nodes (components) according to the specified type. To remove the whole UA object or variable use removeTree().

The NodeIds of the property nodes are generated by the SDK. An application provide own NodeIds by overwriting the method generateNodeIdForSubNode. Default implementation will provide unique NodeIds for the properties generated by SDK.

Parameters:

typeId

The type of node which shall be created. NULL can be passed to use a default node type fitting to the current node

Note:

The node and the sub-nodes are NOT inserted in the address space. Insertion into the address space (becoming visible to a client) will happen only if the node is referenced by another node in the address space or via an explicit "insertTree"

The typeId must only be specified if the node type doesn't match the type of the used class. e.g. if it is a subtype with the same properties or a custom type.

Reimplemented in SoftingOPCToolbox5::Server::DictionaryVariable, SoftingOPCToolbox5::Server::TwoStateItem, SoftingOPCToolbox5::Server::DataItem, SoftingOPCToolbox5::Server::MultiStateItem, SoftingOPCToolbox5::Server::File, SoftingOPCToolbox5::Server::AnalogItem, SoftingOPCToolbox5::Server::MultiStateValueItem, and SoftingOPCToolbox5::Server::Role.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::insert()

Inserts the node into the server's address space. Returns a bad status code in case of an error.

Note:

Only this node is inserted into address space. All other nodes referenced by this node will not be inserted! In most cases insertTree should be used.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::insertTree()

Inserts the given node and all hierarchically referenced nodes into the address space of the server. This method is provided for inserting a complete node tree with a single call.

Note:

If any node within that tree cannot be added (e.g. NodeId already exists) the complete tree will not be added.

Whenever adding a reference via addReference from a node (already residing in the address space) to this node is added, insertTree is automatically invoked.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::remove()

Removes the node from the server's address space.

•All references from the node itself will be removed i.e. a successive call to insertTree() will not insert any (former) child nodes.
•The referenced nodes will stay in address space if they contain any additional references (including e.g. HasTypeDefinition, ...).
•This can lead to islands of nodes in the address space which cannot be reached.
•Any node staying in address space will not be deleted.

Note:

Be careful when removing a single node from address space. This can lead to unexpected leaks. Type nodes can only be removed when neither this type nor any subtype are used.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::removeTree()

Removes the node itself and all nodes referenced by hierarchical references from the server's address space.

Note:

All nodes referenced recursively with any hierarchical reference type are removed. That includes also reference types like HasEventSource or HasNotifier. Type nodes can only be removed when neither this type nor any subtype are used.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::copyTree(ObjectPointer< Server::BaseNode > * copiedTree) const

Copies this node and all hierarchically referenced nodes recursively ("deep copy"). The copied nodes have the same non-hierarchical references as their original nodes. However if the original target node of a non-hierarchical reference has also been copied in the same method call, the same reference will point to the copied target node.

The caller is provided with a pointer to the copied node. Only Objects and Variables can be copied.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::copyAttributeValues(Server::BaseNode * targetNode) const

Copies this node's attributes and its type ("shallow copy"). References (except for the HasTypeDefinition reference) are not copied. A new NodeId will be generated and assigned to the target node.

The caller has to provide a valid pointer to an "empty" target node.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::setBrowseName(const IQualifiedName * browseName)

Sets the human-readable, non-localized browse name of the node. This name is used by clients to create paths out of browse names (see Client::Session::translateBrowsePathToNodeIds()).

Parameters:

browseName

The browse name for the node.

Note:

Two nodes 'A' and 'B' that are referenced by another node 'C' may share the same browse name. However, this is not allowed if the node 'C' is a variable or object type. Thus, browse names must be unique within a type definition but not within an instance of a type.

Since the browse name needs to be parsed at client side (e.g. translateBrowsePathToNodeIds, dictionary XML of data types, ...) browse names containing special characters (e.g. ./"()<>':&) may cause issues at some clients. It is recommended that browse names should contain alpha-numeric characters only including underlines (_) or hyphens (-). Exceptions are instance declarations with ModellingRule OptionalPlaceholder or MandatoryPlaceholder, here it is a common pattern to write the name in angle brackets like <PlaceholderName>, the concrete instances for this placeholder should get concrete names without angle brackets.

EnumStatusCode SoftingOPCToolbox5::Server::BaseNode::setRolePermissions(const std::vector< RolePermissionType > & rolePermissions)

Defines the user access permissions on this node for each role.

The roles to evaluate are bound to the Server::Session, while every ServerSession can have multiple roles. The effective access permissions on a session are calculated by a logical OR of all user permissions of all available session roles and with a logical AND for the common access permission, e.g. from the AccesLevel attribute of Variables or the Executable attribute of Methods.

When any roles are configured on this node, then only the configured roles get the configured access, all not configured roles have no access.

When no role permissions are configured (an empty array), then this has the special meaning to fall back to the default permissions, which grant full user access for every role. This is also the default configuration.

See also:

Server::Session::setRoles()

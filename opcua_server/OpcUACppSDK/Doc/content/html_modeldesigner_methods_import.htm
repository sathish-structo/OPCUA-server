The Import method is for importing an existing Nodeset2 XML file

Input Parameters:

ImportFile

The (path and) filename of the XML file to be imported.

DublicateNodeHandling

The behaviour if the XML file contains nodes which already exist in the address space.

•0 report an error and stop import
•1 means use the existing Node
•2 means use the new Node

Output Parameters:

Status

This shows if the action was successful

Result

A list of errors or warnings occurred while executing the method, or a message that everything is fine.

GlobalExtensions

An array of XMLElements contained in the global "Extension" tags of the imported XML file

 

A "GlobalExtension" XML element typically contains system specific information, which can be used by the importing server application for global settings (e.g. global sampling rate for devices, ...).

It can be a XML tag (including sub-elements) or a simple text.

 

The meta-data of a namespace can be exposed in the address space as a component (type "NamespaceMetadataType" ) of the "NameSpaces" node below the "Server" node.

The OPC UA C++ Server & Client SDK V6.00 does not yet create the "NamespaceMetadataType" object type or the "NameSpaces" node by default. When importing the XML "Opc.Ua.NodeSet2.xml" the nodes are available in address space.

However the OPC UA C++ Server & Client SDK V6.00 does not give any support for setting / extracting meta-data information from the address space; the application has to create the nodes and set their values manually.

 

 

Note:

The DuplicateNodeHandling parameter value "2", (use new node) will not work if the existing node is of a different NodeClass.

E.g. it is NOT possible to replace an existing "Variabe" node with a new "Method" node. The same is valid for ALL NodeClasses.

 

As well the option "1" (use existing node) will cause issues if the NodeClass of any node in the XML is different to the NodeClass of the existing node.

E.g. an existing "VariabelType" node and a new "ObjectType" node can cause problems, since new ObjectType nodes in the XML may be linked via a HasSubType reference to the node in the XML.

The SDK forbids HasSubType references between different type NodeClasses.

 

In general a NodeId shall represent the same "physical" information. So it is not allowed to re-use the same NodeId for a completely different thing.

If you encounter this issue when importing a newer version of a XML, then one of the versions is typically invalid.

OPC UA C++ Server & Client SDK V6.00

Application Configuration File

Top  Previous  Next

In addition to the possibility to configure a Server or Client application through the C++ API (covered in the Introduction to the SDK and Programming Tutorial) the application developer has the possibility to load configuration settings from an external XML configuration file. This approach offers a lot of flexibility, allowing the change of configuration without recompiling the application binaries.

The use of configuration files is demonstrated in the sample for Secure Communication.

 

Enable configuration file usage

The structure of the application configuration XML file and the available configuration elements are described in the next section. In order to enable the use of this configuration file the application developer needs to follow some simple steps:

1. Store the configuration file in a location accessible for the application.
2. Avoid configuring settings that will be loaded from the configuration file (e.g. do not call Application::setPkiStoreConfiguration()), unless it is intended to set or overwrite some specific settings.
3. Initialize the application by using the overloaded method Application::initialize(), providing the correct path to the configuration file and the password for the private key file.
Note: All imported configuration settings except the ApplicationDescription can still be modified or extended by the application code.
4. After a successful call to Application::initialize() the application is configured and initialized. It is now safe to start the application using Application::start().
5. (For Server application) Endpoints are added by the initialization method according to the configuration file. After a successful start of the application you should be able to access the endpoints using Application::getEndpoints() and call Endpoint::open() to make them available to clients.

 

Configuration File Structure

All the available options and settings are grouped under the root element ApplicationConfiguration. The XML configuration file for both Server and Client applications should have the following structure:

<?xml version="1.0" encoding="utf-8"?>

<ApplicationConfiguration

  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

  xmlns:ua="http://opcfoundation.org/UA/2008/02/Types.xsd"

  xmlns="http://opcfoundation.org/UA/SDK/Configuration.xsd"

>

    <!-- Configuration elements in predefined order -->

 

</ApplicationConfiguration>

The supported configuration elements are listed in the following list, in the order they should appear in the configuration file. Some elements are not mandatory, they can be left out or inserted as empty elements. Equivalence between API calls and the configuration elements is presented where possible.

Note: All paths used in the configuration file should be absolute or relative to the current working directory. On VxWorks only absolute paths are accepted.

 

1. ApplicationName [mandatory] - Provide a descriptive name for the application.
2. ApplicationUri [mandatory] - Configures the globally unique identifier for the application instance. This URI shall match to the ApplicationUri in the ApplicationInstanceCertificate and shall be unique among all OPC UA applications.
Note: If the ApplicationUri contains the full name "localhost", the SDK automatically replaces "localhost" by the real hostname. This is useful when using the same config file on several machines. The name "localhost" itself should not appear within the effective ApplicationUri, as it is no global unique identifier.
3. ProductUri [optional] - Configures the globally unique identifier for the product.
4. ApplicationType [mandatory] - Specify the type of the application. Valid options are according to the possible values of EnumApplicationType enumeration:

Server_0

EnumApplicationType_Server

The application is acting as Server only.

Client_1

EnumApplicationType_Client

The application is acting as Client only.

ClientAndServer_2

EnumApplicationType_Both

The application is both Server and Client.

DiscoveryServer_3

EnumApplicationType_DiscoveryServer

The application is acting as a Discovery Server.

6. ServerConfiguration [optional] - Server application specific configuration. This is used only for ApplicationType Server_0 or ClientAndServer_3.
6.1. BaseAddresses [optional] - List of server endpoint URLs.
Note: If a URL uses "localhost" as hostname, then the SDK automatically replaces the hostname with the real computer name. This is useful when you use or copy the same configuration file on several machines. If it is intended to use the local loopback adapter, you should use IP representation "127.0.0.1" for IPv4 or "[::1]" for IPv6.
